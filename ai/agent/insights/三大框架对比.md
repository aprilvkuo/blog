# 三大框架对比

> 目标：理解不同设计哲学，知道什么场景用什么框架

---

## 核心洞察

| 框架 | 设计哲学 | 核心抽象 |
|------|----------|----------|
| LangGraph | 图结构流程 | Node → Edge → StateGraph |
| CrewAI | 角色分工 | Agent → Task → Crew → Process |
| AutoGen | 对话驱动 | ConversableAgent → GroupChat |

**本质差异**：
- LangGraph：**流程控制优先**，适合复杂编排
- CrewAI：**角色抽象优先**，适合固定协作模式
- AutoGen：**对话灵活优先**，适合研究原型

---

## LangGraph

### 设计哲学

Graph-based orchestration

```
┌──────┐    ┌──────┐    ┌──────┐
│ Node │ → │ Node │ → │ Node │
└──────┘    └──────┘    └──────┘
     ↓           ↑
  Condition    Loop
```

### 核心概念

- **Node**：执行单元（可以是 Agent、工具、函数）
- **Edge**：节点间的连接（可以有条件分支）
- **StateGraph**：管理状态流转的图结构
- **State**：在节点间传递的共享状态

### 关键问题

1. **为什么用图而不是链？**
   - 支持条件分支、循环、并行
   - 更复杂流程的可视化
   - 状态管理更清晰

2. **State 在节点间如何传递？**
   - 每个 Node 返回 State 的更新
   - State 是累积的（append）还是替换的（override）
   - 使用 TypedDict 定义 State 结构

3. **Human-in-the-loop 如何实现？**
   - interrupt_before / interrupt_after
   - Command 对象控制流程
   - Checkpointing 持久化状态

### 适用场景

- ✅ 复杂流程控制
- ✅ 可观测性要求高（LangSmith 集成）
- ✅ 生产环境
- ❌ 简单线性流程（过度设计）

## CrewAI

### 设计哲学

Role-based collaboration

```
┌───────────┐
│   Crew    │
├───────────┤
│ Agent 1   │ → Task 1
│ Agent 2   │ → Task 2
│ Agent 3   │ → Task 3
└───────────┤
│  Process  │（sequential / hierarchical）
└───────────┘
```

### 核心概念

- **Agent**：角色定义（goal、role、backstory）
- **Task**：任务定义（description、expected_output）
- **Crew**：Agent + Task 的组合
- **Process**：执行方式（sequential / hierarchical）

### 关键问题

1. **Role 抽象是否过度？**
   - 优点：语义清晰，易于理解
   - 缺点：灵活度受限，自定义困难

2. **Process 类型的权衡？**
   - sequential：简单但固定
   - hierarchical：灵活但复杂（需要 manager agent）

3. **与 LangGraph 的本质区别？**
   - CrewAI：角色抽象层更高，适合固定模式
   - LangGraph：流程抽象层更低，适合自定义

### 适用场景

- ✅ 角色分工明确
- ✅ 协作模式固定
- ✅ 快速原型
- ❌ 复杂流程控制
- ❌ 生产级可靠性

---

## AutoGen

### 设计哲学

Conversation-driven

```
┌──────────────────────────────────────┐
│            GroupChat                  │
├──────────────────────────────────────┤
│  Agent A ← → Agent B ← → Agent C     │
│         ↓                              │
│    ConversableAgent                   │
│    (消息传递、工具调用)                  │
└──────────────────────────────────────┘
```

### 核心概念

- **ConversableAgent**：可对话的 Agent
- **GroupChat**：多 Agent 对话管理
- **Speaker Selection**：发言者选择策略

### 关键问题

1. **对话模式的灵活性 vs 可控性？**
   - 灵活：Agent 自由对话
   - 可控性差：难以预测流程

2. **如何处理对话死循环？**
   - max_round 限制
   - termination condition
   - 但容易误判

3. **研究导向 vs 生产导向？**
   - 研究导向：灵活、可实验
   - 生产就绪度低：可靠性不足

### 适用场景

- ✅ 研究原型
- ✅ 实验新架构
- ✅ 学术论文
- ❌ 生产环境
- ❌ 可靠性要求高

| 维度 | LangGraph | CrewAI | AutoGen |

| **设计哲学** | 图结构流程 | 角色分工 | 对话驱动 |
| **抽象层级** | 低（接近底层） | 高（角色语义） | 中（对话语义） |

| **灵活性** | 高 | 中 | 高 |

| **持久化** | ✅ Checkpointing | ⚠️ 有限 | ❌ 有限 |

## 选择建议

| 场景 | 推荐 | 原因 |
|------|------|------|
| 生产级 Agent 系统 | LangGraph | 可靠性、可观测性、持久化 |
| 快速原型、固定协作 | CrewAI | 语义清晰、上手快 |
| 研究实验、论文 | AutoGen | 灵活、可定制 |
| 简单任务 | 无框架 | 直接 API 更清晰 |
| 特殊需求 | 自建 | 框架抽象不匹配 |

## 关键洞察

### 框架的共同本质

所有框架都实现了：
- Agent = LLM + Tool Call Loop
- 消息历史管理
- 工具定义封装
- 循环控制

### 框架的本质差异

差异在于**抽象层级**和**工程取舍**：
- 抽象越高 → 易用性越好 → 灵活度越低
- 工程越深 → 可靠性越好 → 学习成本越高

### 框架解决不了什么

- Planning 能力（模型层）
- Multi-Agent 协调的本质难题
- Evaluation 和持续改进

## 面试问题

### Q: 什么场景用什么框架？

**回答要点**：
1. 生产级 → LangGraph（可靠性、可观测性）
2. 快速原型 → CrewAI（语义清晰）
3. 研究 → AutoGen（灵活）
4. 简单场景 → 无框架

### Q: LangGraph vs CrewAI 的本质区别？

**回答要点**：
1. LangGraph：流程抽象，接近底层
2. CrewAI：角色抽象，语义层更高
3. LangGraph 更灵活，CrewAI 更易用

### Q: 为什么 AutoGen 不适合生产？

**回答要点**：
1. 对话模式可控性差
2. 持久化、可观测性有限
3. 设计定位是研究而非生产
