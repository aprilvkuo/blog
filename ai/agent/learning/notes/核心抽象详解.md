## 一、Agent = LLM + Tool Call Loop

### 1.1 本质定义

Agent 的核心是一个**循环**：

```
用户输入 → LLM 推理 → 工具调用 → 观察结果 → LLM 推理 → ... → 最终输出
```

这个循环的**灵魂**在于：**LLM 决定何时调用什么工具，以及何时停止**。

### 1.2 最小实现（100 行代码）

```python
import json
from openai import OpenAI

client = OpenAI()

def create_agent(model="gpt-4", tools=None):
    """最小 Agent 实现"""
    messages = []

    def run(user_input: str, max_iterations=10):
        messages.append({"role": "user", "content": user_input})

        for i in range(max_iterations):
            # 1. LLM 推理
            response = client.chat.completions.create(
                model=model,
                messages=messages,
                tools=tools,
                tool_choice="auto"
            )

            message = response.choices[0].message
            messages.append(message)

            # 2. 检查是否调用工具
            if not message.tool_calls:
                # 没有工具调用，返回最终答案
                return message.content

            # 3. 执行工具调用
            for tool_call in message.tool_calls:
                tool_name = tool_call.function.name
                args = json.loads(tool_call.function.arguments)

                # 执行工具
                result = execute_tool(tool_name, args)

                # 4. 将结果加入消息
                messages.append({
                    "role": "tool",
                    "tool_call_id": tool_call.id,
                    "content": str(result)
                })

        return "达到最大迭代次数"

    return {"run": run}

# 工具定义
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "获取指定城市的天气",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "城市名"}
            },
            "required": ["city"]
        }
    }
}]

def execute_tool(name, args):
    if name == "get_weather":
        return f"{args['city']}今天晴天，25°C"
    return f"未知工具: {name}"

# 使用
agent = create_agent(tools=tools)
result = agent.run("北京今天天气怎么样？")
print(result)
```

### 1.3 关键洞察

| 组件 | 作用 | 框架差异 |
|------|------|----------|
| **LLM** | 决策大脑，选择工具、解析结果 | 不同模型推理能力差异大 |
| **Tools** | 能力边界，定义 Agent 能做什么 | 封装方式不同（JSON Schema vs Pydantic） |
| **Loop** | 执行流程，控制迭代次数、超时 | 同步 vs 异步，并行调用 |

### 1.4 框架对比


**本质差异**：
- LangGraph：显式图结构，可控性强
- AutoGen：对话驱动，灵活但难控
- 共同点：底层都是 `LLM + Tool Call Loop`

## 二、Memory 三种模式

### 2.1 概念辨析

| 类型 | 存储位置 | 生命周期 | 典型实现 |
|------|----------|----------|----------|
| **工作记忆** | 对话上下文 (messages) | 单次会话 | `messages` 列表 |
| **短期记忆** | 内存/缓存 | 会话期间 | 向量检索、滑动窗口 |
| **长期记忆** | 外部存储 | 持久化 | 向量数据库、文件系统 |

### 2.2 工作记忆（Working Memory）

**定义**：当前推理立即可用的上下文，即 `messages` 列表。

```python
# 工作记忆就是对话历史
messages = [
    {"role": "system", "content": "你是一个助手"},
    {"role": "user", "content": "帮我查北京天气"},
    {"role": "assistant", "tool_calls": [...]},
    {"role": "tool", "content": "北京今天晴天，25°C"},
    # ...
]
```

**挑战**：上下文窗口有限，需要：
- 截断策略（保留最近 N 条）
- 摘要压缩（定期总结）
- 关键信息提取

### 2.3 短期记忆（Short-term Memory）

**定义**：跨会话共享的临时存储，通常用向量检索。

```python
from langchain.embeddings import OpenAIEmbeddings
from langchain.vectorstores import Chroma

class ShortTermMemory:
    def __init__(self):
        self.embeddings = OpenAIEmbeddings()
        self.vectorstore = Chroma(embedding_function=self.embeddings)

    def add(self, content: str, metadata: dict = None):
        """存储记忆"""
        self.vectorstore.add_texts([content], metadatas=[metadata])

    def search(self, query: str, k: int = 5):
        """检索相关记忆"""
        return self.vectorstore.similarity_search(query, k=k)

    def get_context(self, query: str) -> str:
        """获取增强上下文"""
        docs = self.search(query)
        return "\n".join([doc.page_content for doc in docs])

# 使用
memory = ShortTermMemory()
memory.add("用户偏好：喜欢简洁的回答", {"type": "preference"})
memory.add("上次任务：完成了数据分析报告", {"type": "task"})

# 在新查询时注入相关记忆
context = memory.get_context("帮我写报告")
messages.append({"role": "system", "content": f"相关背景：\n{context}"})
```

### 2.4 长期记忆（Long-term Memory）

**定义**：持久化存储，支持跨会话、跨时间的知识保留。

```python
from datetime import datetime

class LongTermMemory:
    def __init__(self, db_path: str):
        self.db_path = db_path
        self.vectorstore = Chroma(
            persist_directory=db_path,
            embedding_function=OpenAIEmbeddings()
        )

    def save(self, content: str, metadata: dict = None):
        """持久化存储"""
        metadata = metadata or {}
        metadata["timestamp"] = datetime.now().isoformat()
        self.vectorstore.add_texts([content], metadatas=[metadata])

    def recall(self, query: str, time_range=None, k: int = 10):
        """召回历史记忆"""
        # 支持时间过滤
        docs = self.vectorstore.similarity_search(query, k=k)
        if time_range:
            docs = [d for d in docs if self._in_range(d.metadata, time_range)]
        return docs

    def forget(self, condition: dict):
        """遗忘机制"""
        # 实现 TTL 或主动删除
        pass
```

## 三、State Management：有状态 vs 无状态

### 3.1 核心区别

| 特性 | 无状态 Agent | 有状态 Agent |
|------|-------------|-------------|
| **定义** | 每次请求独立，不保存上下文 | 保存会话状态，支持断点续传 |
| **实现** | 单次 API 调用 | 状态机 + 持久化 |
| **适用场景** | 单轮问答、无状态 API | 多轮对话、长任务 |
| **复杂度** | 低 | 高（需要状态管理、并发控制） |

### 3.2 无状态实现

```python
def stateless_agent(user_input: str, tools: list) -> str:
    """无状态 Agent：每次调用独立"""
    response = client.chat.completions.create(
        model="gpt-4",
        messages=[{"role": "user", "content": user_input}],
        tools=tools
    )
    return response.choices[0].message.content

# 每次调用都是独立的
result1 = stateless_agent("你好")  # 不知道之前的对话
result2 = stateless_agent("我刚才说什么？")  # 无法回答
```

### 3.3 有状态实现

```python
from typing import TypedDict, Annotated
from langgraph.graph import StateGraph, END
import operator

# 定义状态结构
class AgentState(TypedDict):
    messages: Annotated[list, operator.add]  # 消息历史，自动追加
    current_task: str                         # 当前任务
    completed_steps: list                     # 已完成步骤
    retry_count: int                          # 重试次数

class StatefulAgent:
    def __init__(self):
        self.graph = self._build_graph()
        self.checkpointer = MemorySaver()  # 状态持久化

    def _build_graph(self):
        graph = StateGraph(AgentState)

        # 添加节点
        graph.add_node("plan", self.plan_node)
        graph.add_node("execute", self.execute_node)
        graph.add_node("reflect", self.reflect_node)

        # 定义边
        graph.set_entry_point("plan")
        graph.add_edge("plan", "execute")
        graph.add_conditional_edges(
            "execute",
            self.should_continue,
            {
                "continue": "execute",
                "reflect": "reflect",
                "end": END
            }
        )

        return graph.compile(checkpointer=self.checkpointer)

    def run(self, input: str, thread_id: str):
        """运行 Agent，支持断点续传"""
        config = {"configurable": {"thread_id": thread_id}}
        return self.graph.invoke(
            {"messages": [{"role": "user", "content": input}]},
            config=config
        )

    def resume(self, thread_id: str):
        """从断点恢复"""
        config = {"configurable": {"thread_id": thread_id}}
        return self.graph.invoke(None, config=config)

# 使用
agent = StatefulAgent()

# 第一次运行，可能中断
result1 = agent.run("分析这个复杂任务...", thread_id="task-001")

# 可以从断点恢复
result2 = agent.resume(thread_id="task-001")
```

### 3.4 状态管理最佳实践

```python
# 1. 状态序列化
import pickle

def save_state(state: AgentState, path: str):
    with open(path, "wb") as f:
        pickle.dump(state, f)

def load_state(path: str) -> AgentState:
    with open(path, "rb") as f:
        return pickle.load(f)

# 2. 状态快照（用于回滚）
def checkpoint(state: AgentState) -> AgentState:
    return state.copy()

# 3. 并发控制
from threading import Lock

class ConcurrentStateManager:
    def __init__(self):
        self.states = {}
        self.locks = {}
        self.global_lock = Lock()

    def get_lock(self, thread_id: str):
        with self.global_lock:
            if thread_id not in self.locks:
                self.locks[thread_id] = Lock()
            return self.locks[thread_id]

    def update(self, thread_id: str, update_fn):
        with self.get_lock(thread_id):
            self.states[thread_id] = update_fn(self.states.get(thread_id))
```

### 3.5 LangGraph 状态管理

```python
from langgraph.checkpoint.memory import MemorySaver
from langgraph.checkpoint.sqlite import SqliteSaver

# 内存持久化（开发用）
checkpointer = MemorySaver()

# SQLite 持久化（生产用）

# 编译时注入
```

### 4.3 工具定义最佳实践

```python
from pydantic import BaseModel, Field
from typing import Optional, List
from enum import Enum

# 1. 使用 Pydantic 做参数验证
class TimeUnit(str, Enum):
    SECOND = "second"
    MINUTE = "minute"
    HOUR = "hour"

class ScheduleTaskInput(BaseModel):
    """安排定时任务的输入参数"""
    task_name: str = Field(
        description="任务名称，简洁明了",
        min_length=1,
        max_length=100
    )
    delay: int = Field(
        description="延迟时间",
        ge=1,
        le=86400  # 最多延迟一天
    )
    unit: TimeUnit = Field(
        default=TimeUnit.MINUTE,
        description="时间单位"
    )
    tags: Optional[List[str]] = Field(
        default=None,
        description="任务标签，用于分类",
        max_items=5
    )

# 2. 清晰的文档字符串
def schedule_task(task_name: str, delay: int, unit: str = "minute", tags: list = None) -> str:
    """
    安排一个定时任务在指定时间后执行。

    Args:
        task_name: 任务名称，用于识别任务
        delay: 延迟时间
        unit: 时间单位 (second/minute/hour)
        tags: 可选的任务标签列表

    Returns:
        任务ID和预计执行时间

    Raises:
        ValueError: 当延迟时间超过限制时
    """
    # 验证逻辑
    if delay <= 0:
        raise ValueError("延迟时间必须为正数")

    # 执行逻辑
    task_id = f"task_{uuid.uuid4().hex[:8]}"
    return f"任务 {task_id} 已安排，将在 {delay} {unit} 后执行"

# 3. 错误处理封装
class ToolResult:
    def __init__(self, success: bool, data: any = None, error: str = None):
        self.success = success
        self.data = data
        self.error = error

    def to_string(self) -> str:
        if self.success:
            return json.dumps({"success": True, "data": self.data})
        return json.dumps({"success": False, "error": self.error})

def safe_tool_execution(tool_fn, *args, **kwargs) -> ToolResult:
    try:
        result = tool_fn(*args, **kwargs)
        return ToolResult(success=True, data=result)
    except Exception as e:
        return ToolResult(success=False, error=str(e))
```

### 4.4 复杂工具设计模式

```python
# 模式1：工具链（一个工具调用另一个工具）
@tool
def analyze_data(data_source: str) -> str:
    """分析数据源并返回洞察"""
    # 先获取数据
    raw_data = fetch_data(data_source)
    # 再处理
    processed = process_data(raw_data)
    # 最后分析
    return generate_insights(processed)

# 模式2：工具组合（返回工具调用建议）
@tool
def plan_analysis(goal: str) -> str:
    """规划分析步骤，返回需要调用的工具列表"""
    steps = [
        {"tool": "fetch_data", "args": {"source": "database"}},
        {"tool": "process_data", "args": {"method": "clean"}},
        {"tool": "generate_insights", "args": {}}
    ]
    return json.dumps(steps)

# 模式3：人机协作工具（需要确认的操作）
@tool
def delete_files(pattern: str, require_confirmation: bool = True) -> str:
    """删除匹配模式的文件（危险操作）"""
    files = glob.glob(pattern)

    if require_confirmation:
        # 返回确认请求，而不是直接执行
        return {
            "status": "confirmation_required",
            "files_to_delete": files,
            "message": f"确认删除 {len(files)} 个文件？"
        }

    # 用户确认后执行
    for f in files:
        os.remove(f)
    return f"已删除 {len(files)} 个文件"
```

## 五、总结：框架选择的本质

| 需求 | 推荐选择 | 理由 |
|------|----------|------|
| 快速原型 | AutoGen / CrewAI | 高度抽象，开发快 |
| 精细控制 | LangGraph | 图结构显式可控 |
| 简单任务 | 直接 API 调用 | 无需框架，最简单 |
| 复杂工作流 | LangGraph | 支持循环、分支、并行 |
| 多 Agent 协作 | AutoGen | 天然支持对话模式 |

**核心认知**：
1. 框架只是工具，本质都是 `LLM + Tool Call Loop`
2. 选择框架取决于你需要多少控制粒度
3. 生产环境优先考虑状态管理、错误处理、可观测性
