# OpenDevin - 大白话版本

> 📄 **原论文**: [OpenHands: An Open Platform for AI Software Developers as Generalist Agents](../pdfs/OpenDevin_2407.16741.pdf)
>
> 🎓 **一句话概括**: 一个让 AI 像程序员一样工作的开源平台

---

## 这个项目是干嘛的？

想象一下，你有一个 AI 助手，它能：
- 打开终端敲命令
- 写代码、改 bug
- 上网查资料
- 还能找其他 AI 同事帮忙

**OpenDevin（后改名 OpenHands）就是这样一个"AI 程序员工位"**——给它一台虚拟电脑，它能像真正程序员一样干活。

---

| 维度 | 信息 |
|------|------|
| **谁** | UIUC、CMU、Yale 等高校 + 阿里巴巴等公司，188+ 贡献者 |

## 为什么需要这个？

### 问题：现有 AI 框架都有短板

| 框架        | 能干啥             | 缺啥          | Stars |
| --------- | --------------- | ----------- | ----- |
| AutoGPT   | 通用任务            | 没安全沙箱，不能写代码 | 170K+ |
| LangChain | 工具调用            | 没浏览器，没安全环境  | 131K+ |
| Deer-Flow | 长周期任务，研究+写代码+创作 | 新项目，生态还在发展  | 50K+  |
| CrewAI    | 角色扮演式多智能体协作     | 没浏览器，没代码沙箱  | 47K+  |
| AutoGen   | 多智能体聊天          | 没浏览器，没代码沙箱  | 56K+  |
| LangGraph | 图结构工作流编排        | 学习曲线较陡      | 28K+  |
| SWE-Agent | 写代码             | 只能写代码，不能上网  | -     |

**一句话**：没有"全能型"AI 开发平台，大家都得自己造轮子。

### 框架 vs 产品 vs 编排工具：深度对比

#### 一、分类定义

| 类型            | 定义                              | 底层技术                               | 代表                                                         |
| ------------- | ------------------------------- | ---------------------------------- | ---------------------------------------------------------- |
| **框架/SDK**    | 提供 API/组件，让你**开发**自己的 Agent     | Python/TypeScript SDK，调用 LLM API   | LangChain, AutoGen, CrewAI, LangGraph, OpenHands SDK       |
| **产品级 Agent** | 开箱即用的 **AI 助手/程序员**，完整工具链       | 自研工具链 + LLM API（Claude/GPT/Gemini） | **Claude Code**, Codex CLI, Cursor, Windsurf, Devin, Manus |
| **编排工具**      | **管理多个 Agent** 并行工作，本身不内置 Agent | Electron/Go，调用外部 CLI               | OpenClaw/AionUi, Claude Squad, Superset                    |

#### 二、Claude Code 属于什么？

**Claude Code = 产品级 Agent**（Anthropic 自研）

| 层面      | 技术/特点                                                           |
| ------- | --------------------------------------------------------------- |
| **定位**  | CLI 编程助手，类似 Cursor Agent 模式                                     |
| **底层**  | Claude Agent SDK（Anthropic 自研）+ Claude 模型                       |
| **工具链** | Task（子 Agent）、Bash、Glob、Read、Edit、Write、Grep、WebFetch、WebSearch |
| **能力**  | 读写文件、执行命令、搜索代码、创建 PR/commit、调用子 Agent                           |
| **特点**  | 极简输出风格、自动任务管理、深度 IDE 集成                                         |

> Claude Code 和 Cursor、Windsurf 属于同一类：**自研工具链 + LLM API**，不依赖 LangChain 等框架。

**产品级 Agent 分两类**：

| 类型 | 特点 | 代表 |
|------|------|------|
| **商业产品** | 闭源、订阅制、官方支持 | Claude Code, Cursor, Windsurf, Devin, Manus |
| **开源产品** | 开源、免费、社区维护 | **Codex CLI**, Crush, Cline, Gemini CLI |

#### 三、用户群体对比

| 类型            | 主要用户                  | 技能要求           | 付费模式                |
| ------------- | --------------------- | -------------- | ------------------- |
| **框架/SDK**    | 开发者、研究者、企业技术团队        | 高（需写代码、懂架构）    | 免费/自托管，LLM API 费用自理 |
| **产品级 Agent** | 程序员、产品经理、创业者、个人开发者    | 低（开箱即用）        | 订阅制（$20-200/月）或按量付费 |
| **编排工具**      | 高级用户、Agent 重度使用者、团队协作 | 中（需配置多个 Agent） | 免费/开源为主             |
|               |                       |                |                     |

#### 四、核心能力对比

| 能力维度      | 框架          | 产品级 Agent   | 编排工具            |
| --------- | ----------- | ----------- | --------------- |
| **定制性**   | ⭐⭐⭐⭐⭐ 完全可控  | ⭐⭐ 有限配置     | ⭐⭐⭐ 调度可配        |
| **上手成本**  | ⭐⭐ 需学习 API  | ⭐⭐⭐⭐⭐ 开箱即用  | ⭐⭐⭐⭐ 简单配置       |
| **功能完整度** | ⭐⭐⭐ 需自己实现   | ⭐⭐⭐⭐⭐ 完整工具链 | ⭐⭐⭐⭐ 多 Agent 协作 |
| **成本控制**  | ⭐⭐⭐⭐ 精细调优   | ⭐⭐ 订阅/按量    | ⭐⭐⭐⭐ 免费/开源      |
| **生产级可用** | ⭐⭐⭐ 需自建基础设施 | ⭐⭐⭐⭐⭐ 直接可用  | ⭐⭐⭐⭐ 稳定编排       |

#### 五、未来发展预测

| 类型 | 2025-2026 趋势 | 挑战 |
|------|----------------|------|
| **框架/SDK** | → **标准化 + 企业化**：LangGraph 成主流，AutoGen/CrewAI 细分领域深耕，企业版收费 | 同质化竞争，差异化难 |
| **产品级 Agent** | → **能力提升 + 价格战**：Claude Code/Cursor/Windsurf 功能趋同，订阅价格下降，免费版增多 | 商业模式竞争激烈，LLM 成本压力大 |
| **编排工具** | → **团队协作 + 企业部署**：多 Agent 编排成标配，团队版/企业版收费，与 CI/CD 集成 | 需绑定特定 Agent 产品，通用性差 |

#### 六、选择建议

| 你的需求             | 推荐选择                                |
| ---------------- | ----------------------------------- |
| 想快速写代码、修 bug     | Claude Code / Cursor / Windsurf     |
| 想研究 Agent 架构、发论文 | LangGraph / AutoGen / OpenHands SDK |
| 想定制企业内部 Agent    | LangChain + 自研工具链                   |
| 需同时跑多个 Agent     | Claude Squad / OpenClaw / Superset  |
| 预算有限、想自己控制       | 开源框架 + 本地 LLM（Ollama）               |

> 💡 **一句话**：**产品拿来干活，框架拿来造轮子，编排拿来管团队**。

#### 七、OpenCode 与 Claude Code 的关系

**OpenCode ≠ 反编译 Claude Code**

| 项目 | 语言 | Stars | 状态 | 来源 |
|------|------|-------|------|------|
| **Claude Code** | TypeScript（闭源） | - | 商业产品 | Anthropic 自研 |
| **Codex CLI** | TypeScript | - | **开源产品** | OpenAI 官方 |
| **OpenCode → Crush** | Go | 11.6K → 仍在更新 | 开源产品 → 已归档，转移给 Charmbracelet | 独立开发 |

**真相**：
- OpenCode/Crush 是**独立开源项目**，使用 Go 语言 + Bubble Tea TUI 框架
- **没有反编译 Claude Code**，而是借鉴了「CLI Agent + 工具链」的设计理念
- 系统提示（System Prompt）可能参考了 Claude Code/Codex CLI 的公开版本

**开源 CLI Agent 对比**：

| 项目 | 开发者 | 特点 |
|------|--------|------|
| **Crush (原 OpenCode)** | Charmbracelet | Go + TUI，多模型支持，LSP/MCP 集成 |
| **Codex CLI** | OpenAI | TypeScript，沙箱+git，apply_patch 编辑 |
| **Cline** | 社区 | VSCode 插件，工具审批机制，详细任务管理 |
| **RooCode** | 社区 | Cline 分支，增强版 |

#### 八、Claude Code 为什么效果好？

**效果好的 5 个关键因素**：

| 因素              | 说明                                                              | 为什么重要                 |
| --------------- | --------------------------------------------------------------- | --------------------- |
| **工具链精良**       | Task（子 Agent）、Bash、Glob、Read、Edit、Write、Grep、WebFetch、WebSearch | 覆盖编程全流程，工具间协作无缝       |
| **极简输出风格**      | 强制 ≤4 行输出，无废话                                                   | 减少噪音，聚焦任务，降低 token 消耗 |
| **自动任务管理**      | TodoWrite + 自动拆解复杂任务                                            | 避免"忘记做事"，进度可见         |
| **子 Agent 机制**  | Task 工具可启动子 Agent 处理子任务                                         | 并行处理，复杂任务不卡住          |
| **Claude 模型本身** | Claude 4.x 的推理能力、代码理解能力                                         | 基础能力强，工具链才能发挥作用       |

**对比 Codex CLI / Cline 的差异**：

| 维度          | Claude Code  | Codex CLI   | Cline                           |
| ----------- | ------------ | ----------- | ------------------------------- |
| **输出风格**    | 极简（≤4 行）     | 详细指导        | 详细指导                            |
| **子 Agent** | ✅ Task 工具    | ❌ 无         | ❌ 无                             |
| **审批机制**    | 隐式           | 显式（用户审批）    | 显式（用户审批）                        |
| **编辑方式**    | Edit 工具      | apply_patch | write_to_file + replace_in_file |
| **任务管理**    | TodoWrite 自动 | 无           | 手动规划                            |

#### 九、框架 SDK 深度对比

| 框架                | 核心优势                 | 主要缺点                 | 适用场景              | 技术栈         |
| ----------------- | -------------------- | -------------------- | ----------------- | ----------- |
| **LangChain**     | 生态最全、组件最多、文档完善       | 过度抽象、学习曲线陡、代码冗长      | 通用 LLM 应用、快速原型    | Python + JS |
| **LangGraph**     | 图结构工作流、状态管理清晰、可视化调试  | 需要理解图概念、代码量较大        | 复杂多步骤流程、企业级 Agent | Python      |
| **AutoGen**       | 多智能体对话天然支持、微软背书      | 单 Agent 能力弱、对话冗长、调试难 | 多 Agent 协作研究、学术项目 | Python      |
| **CrewAI**        | 角色扮演直观、易上手、任务分配清晰    | 工具链不够丰富、定制性有限        | 业务流程自动化、多角色协作     | Python      |
| **OpenHands SDK** | 完整工具链（沙箱+浏览器）、MIT 开源 | 依赖 Docker、部署复杂       | 研究代码型 Agent、二次开发  | Python      |

**架构差异**：

| 框架        | Agent 模型       | 工具调用        | 多 Agent   | 状态管理               |
| --------- | -------------- | ----------- | --------- | ------------------ |
| LangChain | Chain/Runnable | Tool 抽象     | ❌ 需自己实现   | Memory             |
| LangGraph | StateGraph节点   | 绑定到节点       | ✅ 多节点协作   | StateDict          |
| AutoGen   | ConverserAgent | 注册到 Agent   | ✅ 对话协作    | ConversationBuffer |
| CrewAI    | Agent（角色）      | Task + Tool | ✅ Crew 编队 | Task 状态            |
| OpenHands | CodeActAgent   | AgentSkills | ✅ 委托机制    | EventStream        |

**一句话总结**：
- **LangChain** = 组件超市，啥都有但得自己拼
- **LangGraph** = 工程师首选，流程可控可调试
- **AutoGen** = 学术派，多 Agent 对话研究
- **CrewAI** = 产品派，业务流程快速落地
- **OpenHands SDK** = 研究派，代码型 Agent 研究

> 💡 **选择建议**：想快速出活 → CrewAI；想复杂流程 → LangGraph;想研究架构 → OpenHands SDK/AutoGen

> 💡 **注意**：Codex CLI/Gemini CLI/Crush 这类开源产品适合预算有限或想自己控制的用户，但不适合二次开发（代码结构复杂，API 不稳定）。

#### 十、OpenHands vs Claude Code： 谁更有价值？

| 维度 | OpenHands | Claude Code |
|------|-----------|-------------|
| **类型** | 框架/SDK + 产品 | 产品 |
| **开源** | ✅ MIT 开源 | ❌ 闭源 |
| **定制性** | ⭐⭐⭐⭐⭐ 完全可控 | ⭐⭐ 有限配置 |
| **二次开发** | ✅ 可以改源码 | ❌ 不能 |
| **成本** | 自托管，LLM API 自理 | 按量付费 |
| **研究用途** | ✅ 可以发论文 | ❌ 不能 |
| **开箱即用** | ⭐⭐⭐ 需要配置 | ⭐⭐⭐⭐⭐ 直接用 |
| **沙箱环境** | ✅ Docker 完整沙箱 | ❌ 无沙箱（直接操作文件） |
| **浏览器** | ✅ 内置浏览器工具 | ❌ 无浏览器 |

**OpenHands 的独特价值**：

| 场景 | 为什么选 OpenHands 而不是 Claude Code |
|------|--------------------------------------|
| **研究 Agent 架构** | 可以改源码、做实验、发论文 |
| **企业内部定制** | 可以定制工具链、适配内部系统 |
| **安全隔离** | Docker 沙箱，AI 操作不会影响真实系统 |
| **浏览器操作** | Claude Code 没有浏览器工具 |
| **成本控制** | 自托管 + 用便宜的模型 |
| **商用产品** | MIT 协议，可以集成到自己的产品里 |

**Claude Code 的优势**：

| 场景 | 为什么选 Claude Code 而不是 OpenHands |
|------|--------------------------------------|
| **快速写代码** | 开箱即用，不用配置 |
| **日常编程** | 直接在本地操作，不用启动 Docker |
| **子 Agent** | Task 工具很强，可以并行处理 |
| **极简输出** | 不啰嗦，效率高 |
| **IDE 集成** | 深度集成，体验好 |

#### 十一、残酷的现实：框架都很 Toy？

**说实话**：大部分情况下，用产品就够了。

| 对比 | 产品（Claude Code/Manus） | 框架（LangChain/OpenHands） |
|------|--------------------------|---------------------------|
| **工具链** | 专业团队打磨无数遍 | 自己写，很粗糙 |
| **错误处理** | 完善的重试、回退 | 基本没有 |
| **用户体验** | 极简、流畅 | 丑陋、啰嗦 |
| **实际效果** | 真的好用 | Demo 酷，实际拉跨 |
| **迭代速度** | 专业团队每天优化 | 一个人慢慢改 |

**为什么框架显得 Toy？**

| 原因 | 说明 |
|------|------|
| **工具链差距巨大** | Claude Code 有 Task/Bash/Edit/... 完整工具链，你用框架写的只有 `llm.invoke(query)` |
| **过度抽象** | LangChain 写 50 行代码就为调用一个 API |
| **Demo 很酷，实际难用** | 多 Agent 协作在 Demo 里很美好，实际经常卡住、重复工作 |

**框架的真实用户**：

| 用户 | 为什么用框架 |
|------|--------------|
| 研究生 | 要发论文，不能用闭源产品 |
| 企业架构师 | 要定制、要私有化 |
| 创业者 | 要做自己的产品（MIT 协议） |

**一句话**：**产品是精装修的房子，框架是砖头和水泥**——大部分人只想住，不想盖。

> 💡 **建议**：能买现成就别自己造，除非你是造车的工厂。想干活 → 用产品；想研究/发论文 → 用框架但接受效果不如产品。

### 解决思路

> 既然人类程序员用"代码 + 命令行 + 浏览器"这三样工具就能干大部分活，
>
> 那让 AI 也用这三样不就行了？

## 它是怎么工作的？

### 核心架构（3 分钟看懂）

```
┌─────────────────────────────────────────────────────────┐
│                      你（用户）                          │
│                    Web 界面 / 命令行                      │
└─────────────────────┬───────────────────────────────────┘
                      │ 打字说话
                      ▼
┌─────────────────────────────────────────────────────────┐
│                   AI 智能体（大脑）                       │
│         "我要修这个 bug，先看看代码..."                    │
└─────────────────────┬───────────────────────────────────┘
                      │ 发指令
                      ▼
┌─────────────────────────────────────────────────────────┐
│              Docker 沙箱（安全的虚拟电脑）                 │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐               │
│  │ 终端命令  │  │ Python   │  │ 浏览器   │               │
│  │ bash ls  │  │ 代码执行  │  │ 查网页   │               │
│  └──────────┘  └──────────┘  └──────────┘               │
└─────────────────────────────────────────────────────────┘
                      │ 执行结果
                      ▼
              "代码改好了，测试通过！"
```

### 四个关键设计

| 设计 | 通俗解释 | 为什么重要 |
|------|----------|------------|
| **事件流** | 就像聊天记录，所有对话、操作都按时间顺序保存 | AI 能回看自己干了啥，不会失忆 |
| **Docker 沙箱** | 一个隔离的虚拟电脑，AI 在里面随便折腾 | 不会把你真实电脑搞坏 |
| **AgentSkills** | 给 AI 准备的工具箱（改文件、搜代码等） | AI 不用从零学起 |
| **多智能体协作** | 一个 AI 干不了的活，可以委托给专家 AI | 像公司里不同岗位配合 |

## 有什么缺点？

1. **还不够完美**：网页操作成功率才 15%，离人类还差很远
2. **吃资源**：Docker 沙箱启动慢、占内存
3. **长任务容易忘事**：干太久会"失忆"，需要更好的记忆机制
4. **不太会看图**：视觉能力还比较弱
